## Mathlib APIs Inspected at Scaffold Time

| Need | Candidate mathlib module/API | Preliminary decision |
|---|---|---|
| Deterministic partial transition | `Mathlib.Computability.StateTransition`: `eval`, `Reaches`, `Reaches₁`, `Respects` | Reuse or wrap; zero-step semantics already explicit |
| Option-valued partial bijection | `Mathlib.Data.PEquiv`: `PEquiv`, `symm`, `trans` | Strong candidate for reversible one-step systems |
| Set-based partial bijection | `Mathlib.Logic.Equiv.PartialEquiv` | Probably less suitable because functions have irrelevant values outside domains |
| Ordinary TM semantics | `Mathlib.Computability.TuringMachine.PostTuringMachine` (`TM0`, `TM1`, `TM2`) | Inspect deeply before choosing bridge; TM0 splits move and write into separate statements |
| Partial-recursive source | `Mathlib.Computability.Halting`: `ComputablePred.halting_problem` | Preferred established undecidability source |
| Computable reductions | `Mathlib.Computability.Reduce`: `ManyOneReducible`, notation `≤₀` | Preferred reduction framework |
| Words/free monoids | `Mathlib.Algebra.FreeMonoid.Basic`: `FreeMonoid`, `lift`, `of`, `toList` | Reuse |
| Prefix relations | list prefix/suffix APIs in `Mathlib.Data.List.*` | Reuse basic relations; build code-specific lemmas locally |
| Uniquely decodable codes | No dedicated API found in preliminary mathlib source scan | Define a narrow local API using injectivity of an induced `FreeMonoid` homomorphism |

These are source-inspection findings, not final architectural commitments.
Stage 1 must verify exact imports and semantics against the pinned checkout.

## Build Policy

- New declaration skeleton: build the narrow leaf immediately.
- Import change: rebuild the leaf before writing a large proof.
- Shared transition or machine core change: build direct consumers named in
  the stage file.
- API, global instance/notation, or build configuration change: run full
  `lake build`.
- Keep diagnostics and exhaustive probes in `Audit` leaves.
- Avoid importing `Lecerf` from internal modules.

Current scaffold commands:

```text
cd formal
lake update
lake build Lecerf
lake build
```

Scaffold validation on 2026-07-17 generated `lake-manifest.json`; both
`lake build Lecerf` and `lake build` completed successfully with 831 jobs.

## Computability Bridge Risk

Mathlib's halting theorem is phrased for `Nat.Partrec.Code.eval`, while the
paper uses finite quintuple machines. The library must not jump between them by
informal Church–Turing equivalence. Candidate checked routes are:

1. use mathlib's construction from partial-recursive code to a TM model, then
   compile that model into the project's quintuple machine; or
2. formulate the reversible simulation generically for the interpreter
   transition system, prove its reduction first, then separately compile the
   interpreter into the concrete machine.

The chosen route must provide a computable translation and semantic iff, not
just existence of an equivalent machine.
